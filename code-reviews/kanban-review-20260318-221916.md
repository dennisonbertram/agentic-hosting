## CRITICAL

### 1) `UNIQUE(tenant_id)` makes failed boards non-recreatable (and Create’s “failed doesn’t count” check is inconsistent)
**Where**
- Migration: `UNIQUE(tenant_id)` in `kanbans` table
- `Manager.Create`: checks `status != 'failed'`, then always does `INSERT ...`

**Impact**
- If provisioning fails (or `ReconcileStale` marks it failed), the row remains.
- A subsequent `Create()` sees `existingCount == 0` (because status is `failed`) but the `INSERT` will still fail due to `UNIQUE(tenant_id)`.
- Practically: users must manually `DELETE` before they can `CREATE` again; worse, your current retry loop can spin on the same tenant uniqueness error (see HIGH #1).

**Fix options**
- **Best UX:** treat `failed` as “reprovisionable”:
  - Either **delete failed record** up-front (plus best-effort cleanup), then insert new.
  - Or **reuse the existing row** with an UPSERT:
    - `INSERT ... ON CONFLICT(tenant_id) DO UPDATE SET ...` and keep the same `id` (or explicitly rotate it in a safe way).
- If you *want* to allow “one row per tenant ever”, then `Create()` must return a clear conflict that instructs to `DELETE` first, and the API/UX must reflect that. Right now it implies “no board exists” then fails.

### 2) Likely broken provisioning: registration disabled but setup calls `/api/v1/register`
**Where**
- Container env: `VIKUNJA_SERVICE_ENABLEREGISTRATION=false`
- `setupVikunja`: POST `/api/v1/register`

**Impact**
- If Vikunja actually enforces this flag for the register endpoint (likely), setup will fail every time.
- Current behavior marks instance `ready` anyway and returns `adminPassword` as “token” fallback—this may still be unusable if no user was created.

**Fix**
- Confirm Vikunja’s supported bootstrap/admin creation flow and implement that (preferred):
  - Use a supported “initial admin” env var / CLI command / DB seed mechanism if available.
- If the only bootstrap path is registration, then either:
  - Temporarily enable registration in a safe way (but that usually requires restart), or
  - Gate the register endpoint by network exposure (only loopback / only internal) and/or one-time token, then disable.
- **Make setup failure fatal** unless you have a verified alternate way to login.

---

### 3) Kanban containers are not hardened / isolated like other tenant workloads
**Where**
- `Manager.Create` uses `docker.RunDatabase` to run Vikunja
- `docker.RunDatabase`:
  - No `CapDrop`, no `no-new-privileges`, no `ReadonlyRootfs`, no pid/mem limits, no tmpfs hardening
  - Default bridge networking (no per-tenant network isolation); container likely has outbound egress

**Impact**
- A compromise in Vikunja is a much larger blast radius than your service containers.
- In a multi-tenant PaaS, this is a major security regression versus `RunContainer` hardening.

**Fix**
- Don’t use `RunDatabase` for an app server. Create a dedicated `RunKanban`/`RunAppWithVolume` that applies the same hardening as `RunContainer` where possible:
  - `CapDrop: ALL`, `SecurityOpt: no-new-privileges`, `PidsLimit`, resource limits, `ReadonlyRootfs` + tmpfs (and mount only the data dir RW).
- Put kanban containers on an **internal** network and connect ingress proxy (Traefik) rather than relying on host-port mapping (also solves the port-range scalability problem).

---

### 4) Failed/stuck resources can persist indefinitely; GC/reconcile won’t clean them
**Where**
- `ReconcileStale` only handles `status='provisioning'`
- GC only deletes resources **not referenced in DB**
- A failed record can still reference a container_id/volume_name ⇒ GC will not remove it

**Impact**
- If `Create` marks status failed but cleanup didn’t happen (context cancellation, transient Docker failure, crash timing), you can leak containers/volumes forever.
- Also blocks port capacity and consumes disk.

**Fix**
- Extend reconciliation/GC logic:
  - Reconcile `status='failed'` older than X minutes: attempt cleanup and either clear references or delete the record.
  - Or introduce explicit lifecycle states: `provisioning -> ready`, `failed (no resources)` vs `failed_with_resources`, etc.
- Ensure cleanup uses a **fresh background timeout context** so client cancellation doesn’t prevent cleanup.

## HIGH

### 1) UNIQUE constraint retry logic is incorrect (retries on tenant uniqueness conflicts too)
**Where**
- `Create`: retries if `strings.Contains(err.Error(), "UNIQUE constraint failed")`

**Impact**
- On `UNIQUE(tenant_id)` violation, you’ll retry up to 5 times pointlessly, and return a confusing error (“after retries”).
- String matching is brittle across drivers/versions.

**Fix**
- Use sqlite driver error codes (e.g., `sqlite3.Error` with `ExtendedCode`) and distinguish:
  - port uniqueness conflict ⇒ retry
  - tenant uniqueness conflict ⇒ return `apierr.Conflict(...)`

---

### 2) `containerName := tenantID[:8]` can panic
**Where**
- `Create`: `tenantID[:8]`

**Impact**
- Any tenantID shorter than 8 bytes crashes provisioning (panic).

**Fix**
- Use safe slicing:
  ```go
  prefix := tenantID
  if len(prefix) > 8 { prefix = prefix[:8] }
  ```
- Also consider sanitizing tenantID for Docker name constraints.

---

### 3) Health check is too permissive (`<500` counts as healthy) and ignores request context
**Where**
- `waitForHTTP`: `if resp.StatusCode < 500 { return true }`
- No `ctx.Done()` handling

**Impact**
- 404/401/etc can mark a board “healthy”.
- Provisioning continues even if client disconnected or server shutting down.

**Fix**
- Require `200` and ideally validate response JSON.
- Accept `ctx` and abort early on cancellation:
  ```go
  func (m *Manager) waitForHTTP(ctx context.Context, ...) bool { ... select { case <-ctx.Done(): return false ... } }
  ```

---

### 4) `findFreePort` ignores scan errors and doesn’t use context
**Where**
- `rows.Scan(&p)` return value ignored
- `m.db.Query(...)` without context

**Impact**
- Silent DB/scan failure can lead to bad port reuse decisions.
- No cancellation/timeouts on that query.

**Fix**
- Use `QueryContext` and handle scan errors:
  ```go
  if err := rows.Scan(&p); err != nil { return 0, err }
  ```

---

### 5) Encryption failure is ignored but instance is marked ready
**Where**
- `Create`: encrypt token error is logged but provisioning proceeds to `ready`

**Impact**
- `GET /v1/kanban/admin-token` will later 404 permanently.
- You’ve created a “ready but irrecoverable credentials” state.

**Fix**
- Treat inability to store admin credential as fatal:
  - mark failed + cleanup, or
  - store a recoverable fallback (encrypted admin password) in a separate field.
- Also: don’t conflate “admin_token” with “admin_password” (see MEDIUM #2).

---

### 6) Supply-chain and operability risk: `vikunja/vikunja:latest`
**Impact**
- Breaking API changes (your setup endpoints may change)
- Unreviewed image updates in prod

**Fix**
- Pin a known-good version and ideally a digest:
  - `vikunja/vikunja:0.x.y@sha256:...`

**Impact**

**Fix**

### 2) API/field semantics: `admin_token` sometimes returns a password

**Impact**

**Fix**
