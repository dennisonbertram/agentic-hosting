## What Was Done

1. **Local validation passed** — `go build ./...` and `go test ./...` both passed locally before attempting any deployment.

2. **Cross-compile failed (CGO issue)** — Attempted `CGO_ENABLED=0 GOOS=linux go build ./cmd/ah` for a static Linux binary. The build succeeded locally but the resulting binary crashed on the server because `go-sqlite3` requires CGO (it wraps the C SQLite library). This is a fundamental constraint: you cannot cross-compile this binary without a C cross-toolchain set up. Do not attempt `CGO_ENABLED=0` for this project.

3. **Resolution: build on server** — Installed Go 1.25 on the server at `/usr/local/go`. Used `rsync` to copy the source tree to the server, then built with CGO enabled directly on the server. Build succeeded.

4. **Service failed to start: missing bootstrap token** — The new binary requires `AH_BOOTSTRAP_TOKEN` to be set in the environment even in `--dev` mode (unless `--open-registration` is also passed). The existing systemd unit had no `/etc/default/paasd` env file, so the variable was absent. Fixed by creating `/etc/default/paasd` containing a freshly generated 64-character hex token.

5. **Port 8080 conflict** — Service still failed after the env fix because port 8080 was already in use. Investigation revealed the OLD `ah` binary was still running as PID 299680 from a prior manual session — it had never been cleaned up. Killed the orphan process, then the new service started successfully.

6. **Migrations ran cleanly** — Three new migrations applied without error:
   - `state_009_circuit_recovery`
   - `state_010_kanbans`
   - `state_010_snapshots`

7. **Website rebuild** — The website container did not automatically pick up the new `website/dashboard/` files. Rebuilt the container and restarted it with the same Traefik labels. Dashboard now serving correctly.

8. **Systemd enable** — `paasd.service` was set to `disabled` (would not survive a reboot). Ran `systemctl enable paasd` to correct this.

## What Was Learned

- **CGO is a hard requirement** — `go-sqlite3` uses cgo. There is no workaround without switching to a pure-Go SQLite driver (e.g. `modernc.org/sqlite`). The deploy workflow must build on the server or use a proper Linux cross-compilation toolchain with `CC=x86_64-linux-musl-gcc`.
- **Always check for orphan processes before restarting a service** — A prior manual `./ah` invocation was holding port 8080. `lsof -i :8080` or `ss -tlnp | grep 8080` should be part of the pre-deploy checklist.
- **Environment file is required** — The systemd unit uses `EnvironmentFile=/etc/default/paasd`. If that file doesn't exist, the service silently starts without required env vars. This file must be provisioned as part of initial server setup, not left implicit.
- **`systemctl enable` is separate from `systemctl start`** — The service was running but not enabled. A server reboot would have silently lost the service.
- **Website container is not auto-rebuilt** — Static file changes to `website/` are not picked up until the container is explicitly rebuilt and restarted. This should be part of the release checklist.
